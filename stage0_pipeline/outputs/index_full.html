正式版整合

把调试用的 scripts_m2/semantic_transfer_v2.py 整理成正式模块 scripts_m2/color_reference_transfer.py，提供统一命令行： 
python color_reference_transfer.py --ref REF.jpg --tgt TGT1.jpg TGT2.jpg \
    --out-dir OUT/ --strength {light,medium,strong} \
    [--profile-out profile.json]   # 保存参考图分析结果，之后可复用
    [--profile-in profile.json]    # 跳过重新分析参考图，直接套用

Phase A-1：参考图 Style Profile 缓存

以前每套一次参考图色调都要重新对参考图跑一次分割。现在参考图只分析一次，存成 JSON（每个识别到的类别记录 Lab 均值/标准差/亮度分位），之后可以直接加载复用到任意数量的目标照片，不用重复跑分割模型。冒烟测试：先保存 profile，再用 --profile-in 直接套用到新照片，行为一致。

Phase A-2：强度档位

强度从"手改数字"变成三档可选，各档位在同一张图上的效果：

light → medium → strong：天空饱和度和通透度递增，medium 是 Phase B 已验证过的强度，strong 更激进但目前只做了小范围测试，量产前建议先用 medium。

Phase A-3：整图内容匹配度门槛

过程中真的抓到一个新 bug：最初这道门槛只是"打印警告"，没有真正拦截。用一张内容明显不匹配的参考图（户外蓝天）套到一张室内暖色人像照片上时，"clothing"类别的标签虽然同名，但参考图里的衣服和目标照片里的衣服根本不是一回事——被强行迁移后，女士的黑红色外套变成了诡异的白色鬼影。

↑ 修复前：门槛只是警告，衣服被错误染白

修复：把内容匹配度门槛改成硬拦截——整图判定"不适合"时，除了 skin（人脸检测来源，有色相锁保护）和 neutral（本身已有动态收力保护）之外，其余所有具体类别一律跳过迁移，保持"宁可不动，不动错"。

↑ 修复后：内容不匹配时基本保持原样

Phase C：GPT 蒸馏尝试

结论：GPT Image 2 参考仿色接口（双图 images/edits）目前不可靠，Phase C 蒸馏工作被此阻塞，不是本地算法的问题。

具体发现：
  1. 接口经常超时：连续 3 次调用参考仿色接口（传目标图+参考图两张图），全部在代理层 120 秒硬超时失败（Cloudflare Error 524），换小图也一样。
  2. 更严重的问题——历史结果内容对不上：检查之前跑过的两个参考仿色结果文件时发现，它们的画面内容和输入完全不符——本该是"雾中山间牌坊"场景的调色结果，实际存的却是"商场排队"和"K-pop 舞台跳舞"这种完全不相关的画面。说明这个 API 的双图参考编辑功能不仅慢，偶尔返回时内容也可能是错的/不相关的，不能直接当作可信的 teacher 信号使用。
  3. 对照实验：单图编辑（不带参考图）是可靠的——同一批早期测试里，不带参考图的单图编辑结果内容是对的（同一张操场升旗照，只是颜色变化）。所以问题精确定位在"双图参考"这个具体能力上，不是这个 API 整体不能用。今天重新验证单图编辑时，接口目前处于繁忙/超时状态，暂时也没能拿到新的可信 teacher 样本。

问题证据：调色结果画面完全对不上原图

最右侧 "gpt teacher" 面板应该是左边"雾中山间牌坊"照片的调色版本，但实际内容完全是另外的场景——这两个文件不能用于量化残差分析。

建议：Phase C 暂缓，等 API 稳定或确认这家供应商双图参考编辑功能可用后再重新尝试；或者考虑换一种蒸馏思路——不依赖"目标图+参考图"双图输入，改成把参考图的颜色统计量（我们本地算法已经能提取的 Lab 均值/标准差）转成文字描述，用可靠的单图编辑 + 文字描述做蒸馏 teacher，绕开双图接口。

Phase E / F 现状说明

没有开始，也不建议在当前会话里"顺手做完"：
  • Phase E（2000-5000 张伪标签蒸馏 + 训练学生规划模型）是数天到数周量级的数据标注+GPU训练工作，需要单独规划算力和时间预算，不是能在当前会话里现算出来的。
